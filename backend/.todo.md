# Marketly Backend Roadmap

This is the working roadmap for Marketly. It is written as both a task list and a learning guide.

The product goal is not "call GPT and get a stock score." The product goal is:

> Build a structured finance reasoning engine that prepares evidence, understands context, admits uncertainty, and then uses GPT to explain the result clearly.

## Current Backend Shape

```text
backend/
├── app/
│   ├── core/             config, cache, errors, symbol normalization
│   ├── integrations/     Finnhub, FMP, FRED, OpenAI, news providers
│   ├── routes/           FastAPI endpoints
│   ├── schemas/          response contracts
│   ├── services/         finance and analysis logic
│   ├── models.py         internal typed data shape
│   └── serialization.py  JSON-safe cleanup
├── tests/                unit and integration-style tests
├── ARCHITECTURE.md       system guide
├── .todo.md              roadmap
├── pyproject.toml
├── requirements.txt
└── run.py
```

## How To Think About The System

Marketly is a layered analysis pipeline.

```text
Raw data
  -> facts
  -> metrics
  -> signals
  -> interpretation
  -> scenarios
  -> final narrative
```

Each layer has a different job.

Raw data answers:

> What did the provider send us?

Facts answer:

> What do we believe is true?

Metrics answer:

> What does the math say?

Signals answer:

> What does the math imply?

Interpretation answers:

> Why does that matter for this kind of company?

Scenarios answer:

> What futures are plausible?

Final narrative answers:

> So what should the user understand?

## Near-Term Priorities

The order matters. Do not jump to fancy features before the analysis engine is trustworthy.

1. Make the score deterministic.
2. Improve financial metric correctness.
3. Add cash-flow and capital-structure analysis.
4. Improve macro and market context.
5. Improve news/event intelligence.
6. Add Supabase persistence.
7. Add regression fixtures and observability.

## 1. Make The Final Numeric Score Deterministic

### Problem

The final score is currently produced by GPT.

That is useful for prototyping, but it is not ideal for a finance product. A user may ask:

> Why did this stock get 73 instead of 81?

If GPT invented the score, the answer is hard to audit.

### Target design

Compute category subscores in code:

```text
profitability     0-17
growth            0-17
valuation         0-17
balance sheet     0-17
market/news       0-16
macro regime      0-16
```

Then:

```python
final_score = (
    profitability_score
    + growth_score
    + valuation_score
    + balance_sheet_score
    + market_news_score
    + macro_score
)
```

GPT should explain the score, not invent it.

### Why this works

- deterministic scores are testable
- score drift is easier to detect
- users can understand category penalties
- Supabase can store every subscore separately

### Tradeoff

The scoring formula will be imperfect at first. That is okay. A visible imperfect formula is better than an invisible changing score.

### Implementation direction

Add a service:

```text
app/services/scoring/composite.py
```

Example shape:

```python
def build_composite_score(
    scoring_metrics: dict,
    interpretation: dict,
    market_context: dict,
    event_layer: dict,
    data_quality: dict,
) -> dict:
    return {
        "score": 74,
        "subscores": {
            "profitability": 14,
            "growth": 10,
            "valuation": 8,
            "balanceSheet": 13,
            "marketNews": 12,
            "macro": 11,
        },
        "penalties": ["High valuation dependency"],
        "confidenceAdjustment": -3,
    }
```

Then pass this into GPT.

## 2. Fix And Harden Metric Correctness

### Problem

Bad math destroys trust faster than weak narrative.

Examples:

- percent values may arrive as `42` or `0.42`
- PEG can become absurd if growth is treated as decimal in one place and percent in another
- market cap may be in millions from one provider and raw dollars from another

### Why this matters

A summary can sound intelligent while the numeric score is wrong. Users will forgive missing context before they forgive wrong math.

### Current relevant files

```text
app/services/scoring/valuation.py
app/services/scoring/growth.py
app/services/scoring/profitability.py
app/services/scoring/stability.py
app/services/scoring/metrics.py
app/integrations/financials.py
```

### Implementation direction

Add sanity validation before metrics are trusted:

```python
def is_plausible_pe(value: Any) -> bool:
    return isinstance(value, (int, float)) and 0 < value < 500

def normalize_percent(value: Any) -> float | None:
    if value is None:
        return None
    if 1 < abs(value) <= 100:
        return value / 100
    return value
```

Add tests for:

- gross margin as `42` vs `0.42`
- ROE as `28` vs `0.28`
- market cap in millions vs dollars
- PEG with negative growth
- missing EPS

### Tradeoff

Normalization rules can accidentally "fix" a valid edge case incorrectly. That is why implausible values should be flagged, not silently rewritten everywhere.

## 3. Add Cash-Flow And Capital-Structure Scoring

### Problem

The app fetches cash-flow data but does not use enough of it.

Profit can be accounting. Cash flow is harder to fake.

### Concepts

Free cash flow:

```text
cash left after operating needs and capex
```

FCF margin:

```text
free cash flow / revenue
```

FCF conversion:

```text
free cash flow / net income
```

Capex intensity:

```text
capital expenditures / revenue
```

### Why this matters

Two companies can have the same net income but very different quality.

Company A:

- high net income
- strong free cash flow
- low capex

Company B:

- high net income
- weak free cash flow
- heavy capex

Company A is usually higher quality.

### Implementation direction

Add:

```text
app/services/scoring/cash_flow.py
```

Example:

```python
def calculate_fcf_margin(free_cash_flow, revenue):
    if revenue in (None, 0) or free_cash_flow is None:
        return None
    return free_cash_flow / revenue
```

Then add a `cashFlow` block to the response.

### Tradeoff

Cash-flow metrics can be lumpy. A single year of weak FCF does not always mean the business is bad. Interpret cash flow with history and business model.

## 4. Improve Capital Structure Analysis

### Problem

Current stability metrics are useful but basic.

Debt-to-equity alone can mislead:

- buybacks can shrink equity and make debt/equity look worse
- cash-rich companies can carry debt safely
- short-term debt is riskier than long-term debt

### Better questions

- Is net debt rising?
- Is interest coverage healthy?
- Is debt short-term or long-term?
- Does free cash flow cover obligations?
- Does leverage limit flexibility?

### Implementation direction

Extend `app/services/scoring/stability.py`.

Potential output:

```json
{
  "debtToEquity": 1.1,
  "netDebt": 12000000000,
  "interestCoverage": 8.4,
  "leverageRisk": "moderate",
  "capitalStructureNotes": [
    "Interest coverage is healthy.",
    "Debt load is meaningful but cash generation offsets risk."
  ]
}
```

## 5. Improve Market Context

### Current state

`app/services/market_context.py` summarizes:

- S&P 500 trend
- Fed funds direction
- 10Y yield direction
- inflation direction
- company news sentiment
- risk-on/risk-off
- beta sensitivity

### Why this exists

Stocks are priced inside regimes.

High valuation can be tolerated in:

- falling-rate environments
- strong risk appetite
- AI/product narrative markets
- liquidity-driven rallies

The same valuation can be punished in:

- rising-rate environments
- weak index trend
- earnings recession
- risk-off markets

### Next improvements

Add:

- yield curve slope
- unemployment trend
- GDP trend
- sector trend if sector ETF data is available
- market breadth if available
- credit spread proxy if available

### Tradeoff

Do not overbuild macro too early. The market context layer should be a useful input to scenarios, not a fake precision trading model.

## 6. Improve News And Event Intelligence

### Current state

`app/services/events/service.py` uses keyword logic to classify catalysts.

That is transparent and easy to test, but limited.

### Target categories

```text
earnings
guidance
product launch
regulation/legal
M&A
layoffs/restructuring
supply chain
analyst upgrade/downgrade
capital return
management change
clinical/regulatory event
```

### Why this matters

News should affect the analysis through structured signals, not loose article dumps.

Bad:

```text
Here are 20 headlines. GPT, figure it out.
```

Better:

```json
{
  "sentimentSkew": "positive",
  "topCatalysts": [
    {
      "type": "product_launch",
      "importance": "high",
      "tone": "positive",
      "reason": "Launch cycle can change near-term revenue expectations."
    }
  ],
  "risks": ["Delay risk remains material."]
}
```

### Implementation direction

Keep deterministic keyword classification first. Later, add a GPT-assisted or embeddings-assisted summarizer only after deterministic categories exist.

## 7. Add Calendar And Upcoming Event Awareness

### Problem

Some major risks are not visible in recent headlines.

Examples:

- next earnings date
- ex-dividend date
- product event
- FDA decision
- legal ruling
- lockup expiration

### Why this matters

Scenarios need catalysts and clocks.

A stock may be low-risk for five years but high-risk next week because earnings are coming.

### Implementation direction

Add an event calendar layer:

```text
app/services/calendar/service.py
```

Potential output:

```json
{
  "upcomingEvents": [
    {
      "type": "earnings",
      "date": "2026-07-24",
      "importance": "high",
      "whyItMatters": "Guidance can reset growth expectations."
    }
  ]
}
```

## 8. Improve Scenario Calibration

### Current state

Scenarios are GPT-generated from deterministic signals, validated, normalized, and backed by deterministic fallback.

This is much better than hardcoded scenarios, but probabilities are still analyst judgment.

### Next improvement

Create probability bands from deterministic signals before GPT writes scenarios.

Example:

```python
if valuation_richness == "rich" and growth_label == "weak":
    bear_floor = 0.25

if market_context["equityRiskSentiment"] == "risk_on" and beta_sensitivity == "high":
    upside_boost = 0.1
```

Then GPT works inside those bounds.

### Why this matters

GPT should name and explain scenarios. The backend should constrain probabilities.

### Tradeoff

Hard bounds can be too rigid. Use bands, not exact probabilities.

## 9. Improve GPT Prompt Payload Quality

### Current state

GPT receives a mix of raw data and structured layers.

### Target

GPT should receive mostly curated evidence:

- deterministic score/subscores
- data quality
- metrics
- interpretation
- market context
- scenarios
- key catalysts

Raw statements should be fallback context, not the main payload.

### Why this matters

GPT performs better with compact structured evidence than with huge raw blobs.

Bad prompt shape:

```text
Here is everything. Analyze it.
```

Good prompt shape:

```text
Here are the computed facts, confidence levels, scenario cases, and unresolved unknowns. Explain the investment view.
```

## 10. Add Supabase Persistence

### Why this should come after the analysis shape

Databases freeze decisions.

If the analysis object is unstable, Supabase integration becomes churn. The backend now has a better persistence-ready shape because it includes:

- `analysisId`
- `analysisVersion`
- `dataTimestamp`
- provenance
- refresh policy
- layer partitions

### Suggested table direction

```text
analysis_runs
  id
  symbol
  analysis_version
  data_timestamp
  score
  confidence_level
  analysis_source

analysis_metadata
  analysis_run_id
  data_quality_score
  provenance
  refresh_policy
  input_partitions

financial_facts
  symbol
  fact_key
  value
  source
  confidence
  observed_at

computed_metrics
  analysis_run_id
  category
  payload

scenario_cases
  analysis_run_id
  name
  probability
  confidence
  thesis
  probability_rationale
  key_evidence
  watchlist_triggers

  symbol
  provider
  headline
  summary
  published_at
  url

  snapshot_date
  payload
```

### Important design rule

Do not store only the final GPT summary.

Store enough intermediate evidence that the summary can be regenerated later.

## 11. Add Regression Fixtures

### Problem

As the engine gets smarter, it can accidentally get worse.

### Target fixture set

Use different business types:

- Apple: hardware ecosystem
- Microsoft: cloud/software
- Nvidia: high-growth semiconductor/platform narrative
- Tesla: high-beta narrative/manufacturing hybrid
- Take-Two: IP-driven launch cycle
- JPMorgan or Bank of America: bank/rates/credit sensitivity
- a biotech: binary catalyst risk
- a cyclical industrial: macro sensitivity
- a weak company: low quality / high risk

### What to assert

Not exact GPT text. Assert structure and broad direction:

```python
assert result["businessModel"]["primaryModel"] == "hardware_ecosystem"
assert result["analysisMetadata"]["confidenceLevel"] in {"medium", "high"}
assert len(result["scenarios"]["cases"]) >= 3
assert sum(case["probability"] for case in result["scenarios"]["cases"]) == pytest.approx(1.0)
```

## 12. Improve Observability

### Problem

Provider failures are too quiet.

### Needed visibility

For provider calls:

- provider name
- endpoint
- status code
- symbol
- failure reason
- whether fallback was used

For analysis:

- analysis version
- scenario source
- GPT source/fallback source
- data quality score
- missing critical fields

### Why this matters

When a user says "why is this score weird?" you need to answer:

> Was the company weird, or was the data weird?

## Rule Of Thumb For Every New Feature

Before adding code, ask:

1. Is this a fact, metric, signal, interpretation, scenario, or narrative?
2. Can it be calculated deterministically?
3. Should GPT explain it or generate it?
4. How do we know if the input is missing or low quality?
5. Should this be stored in Supabase?
6. What would make this output wrong?

If you can answer those questions, the implementation will usually be straightforward.
