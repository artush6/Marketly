# Marketly Backend TODO

This file tracks the next round of backend work for Marketly.

The ordering below is based on project impact:
- first, fix anything that makes the score wrong
- then, make scoring deterministic and trustworthy
- then, expand context with macro, news, events, and broader search
- finally, improve observability, coverage reporting, and ops

## 1. Fix score correctness and metric miscalculations

This is the highest-priority work because incorrect math directly damages the product's core output.

### Why this matters

- A strong company can receive an artificially weak score
- Summaries can sound reasonable while the numeric score is still wrong
- Users will lose trust faster from bad math than from missing extra context

### Required changes

- Fix PEG calculation in [`app/services/scoring/valuation.py`](/Users/artush/Developer/Marketly/backend/app/services/scoring/valuation.py)
  - make sure earnings growth is handled consistently as percent vs decimal
  - avoid obviously impossible values like `PEG = 140` for a large-cap with moderate growth
- Normalize units and scales across providers
  - `marketCap`
  - `roe`
  - `grossMargin`
  - `dividendYield`
  - any percent-like vs ratio-like field
- Prefer statement-derived or trusted-provider values when summary metrics are ambiguous
- Add sanity checks before metrics are used in scoring
  - reject or flag implausible values
  - detect scaling mismatches
  - detect conflicting provider values when material

## 2. Add financial data validation and coverage guards

The scoring pipeline should know when its inputs are incomplete or suspicious.

### Why this matters

- Missing or malformed data currently degrades the score too silently
- Weak data quality can masquerade as weak fundamentals

### Required changes

- Add a validation layer before scoring
  - verify statement blocks are present when expected
  - verify core fields are non-null and plausible
  - flag missing critical inputs
- Add explicit data-quality metadata
  - overall coverage
  - critical missing fields
  - provider usage
  - freshness where available
- Surface quality warnings in `/score`
  - `valuation metrics partially missing`
  - `balance sheet incomplete`
  - `quote fields incomplete`

## 3. Make the final numeric score deterministic

The numeric score should be computed by code, not mainly by GPT.

### Why this matters

- Deterministic scoring is easier to debug, test, and trust
- GPT is good at explanation, not stable raw scoring

### Target design

Split the pipeline into:
- deterministic subscores
  - profitability
  - growth
  - valuation
  - stability / balance sheet
  - macro regime
  - news / events
- GPT explanation layer
  - summary
  - positives
  - negatives
  - optional bounded adjustment only if needed

### Required changes

- Build explicit numeric weights per score category
- Compute the final score in code
- Let GPT explain the score rather than invent it
- Add confidence scoring tied to data quality and coverage

## 5. Expand financial scoring with cash flow and capital structure

We now fetch more data than we currently score.

### Why this matters

- Cash flow is one of the strongest signals for equity quality
- Capital structure details can heavily change the risk view

### Required changes

- Add cash-flow-driven metrics

## 6. Add structured macro regime analysis

Macro should affect the score through a compact deterministic summary, not only raw data blobs.

### Why this matters

- Raw time series are too noisy for direct model use
- Macro conditions can materially change how valuation and risk should be interpreted

### Required changes

- Add a macro summarizer service
- Produce a compact regime block such as:
  - inflation trend
  - policy-rate trend
  - unemployment trend
  - GDP trend
  - yield-curve regime
  - credit / liquidity pressure
  - consumer-demand pressure
  - USD strength where relevant
- Feed the macro summary into the scoring engine and GPT explanation

## 7. Add structured news and event intelligence

News should be summarized into signals, not passed as a loose article dump.

### Why this matters

- GPT can inconsistently weight raw articles

### Required changes

- Classify stories into categories:

  - guidance
  - product launches
  - regulation / legal
  - M&A
  - layoffs / restructuring

- Extract:
  - positive catalysts

## 10. Improve provider observability and fallback behavior

The current provider layer is resilient, but too quiet when something is wrong.

### Why this matters

- `safe_get(...)` intentionally fails softly
- that is good for uptime, but bad for spotting bad endpoints and silent data loss

### Required changes

- Improve request failure logging

  - endpoint
  - status code where available
  - failure reason
- Track source quality per metric or block where helpful
- Make fallback order more explicit
  - trusted provider metric first
  - computed fallback second
  - lower-confidence fallback last
