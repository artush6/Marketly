# Marketly AI Financial Intelligence Redesign

## Goal

Transform Marketly from a shallow data summarizer into a multi-layer financial intelligence system that:

- reasons under missing data instead of failing
- adapts analysis by business model
- incorporates historical context and event intelligence
- produces scenario-based, conviction-weighted outputs
- adapts recommendations to the user profile
- remains useful when upstream APIs partially fail

## Current Problems Identified

- Current backend flow is too linear: financials + macro + news -> single GPT scoring pass -> summary.
- Follow-up assistant is constrained to the supplied payload and therefore cannot reason beyond missing fields.
- The system treats all companies with a mostly uniform framework.
- Missing data leaks directly into the user-facing experience as "Data missing".
- No persistent historical/event memory exists for product cycles, launch analogs, or engagement trends.
- No explicit uncertainty model or evidence provenance exists.
- No distinction between observed facts, inferred estimates, and low-confidence assumptions.

## Target Architecture

### Layer 1: Data Layer

Purpose:
- collect, normalize, reconcile, and cache raw company data

Inputs:
- financial statements
- quote and market data
- analyst data
- company news
- macro indicators
- historical price data
- earnings transcripts / SEC filings
- event and engagement proxies for specific business models

Outputs:
- canonical normalized fact graph
- provenance and source quality metadata
- coverage and confidence report

Planned modules:
- `backend/app/services/facts/`
- `backend/app/services/history/`
- `backend/app/services/events/`

Key design rules:
- never trust one provider as the only truth source
- store source, timestamp, quality score, and confidence for every fact
- keep "last known good" snapshots for degraded-mode operation

### Layer 2: Interpretation Layer

Purpose:
- convert raw metrics into business meaning

Examples:
- high gross margin does not always imply durable quality
- negative ROE can be misleading if accounting/equity base is distorted
- lumpy revenue plus high-margin IP can still be attractive if monetization tail is durable

Outputs:
- margin quality
- growth durability
- capital intensity
- balance sheet risk
- valuation dependency
- critical unknowns

Implementation style:
- deterministic derived metrics in code
- rule-based interpretation primitives
- LLM used after feature extraction, not before

### Layer 3: Business Model Layer

Purpose:
- classify companies into the correct analytical framework

Core labels:
- `saas`
- `cloud`
- `hardware_ecosystem`
- `manufacturing`
- `ip_driven`
- `live_services`
- `biotech`
- `consumer_platform`
- `cyclical`

Approach:
- compute structured features in code
- apply explicit rules
- use LLM as structured adjudicator
- allow multi-label output with confidence
- allow manual override layer for known edge cases / important tickers

Example:
- TTWO should likely classify as:
  - primary: `ip_driven`
  - secondary: `live_services`

Important principle:
- we are not manually classifying each company every time
- we are manually designing the framework and rules, then letting the system apply them

### Layer 4: Event & Catalyst Layer

Purpose:
- incorporate real-world drivers instead of only static financial snapshots

Must support:
- product launches
- content cycles
- release delays
- regulation
- macro conditions
- competition shifts
- user behavior changes

For IP-driven / gaming names:
- launch timing
- review reception
- online/live-service conversion
- retention risk
- content cadence
- decay curves

Must support lifecycle modeling:
- launch -> hype -> peak -> decay -> stabilization

### Layer 5: Scenario Engine

Purpose:
- generate structured bull/base/bear cases from operational drivers

Outputs per scenario:
- key assumptions
- operational driver tree
- probability
- confidence
- revenue / margin / cash flow implications
- stock implication
- invalidation triggers

For TTWO / GTA VI style names:
- launch quality
- launch timing
- premium unit sales
- online conversion
- retention
- recurrent spend
- post-launch decay rate

### Layer 6: Output Layer

Purpose:
- produce decision-oriented, high-conviction outputs tailored to the user

Must always include:
- conviction level
- asymmetry
- what must go right
- what can go wrong
- observed vs inferred distinction
- confidence / evidence quality

User profile adaptation:
- long-term investor:
  - focus on compounding, moat durability, franchise longevity, reinvestment runway
- short-term trader:
  - focus on catalyst timing, expectations, volatility, positioning risk

## Business Model Analysis Design

Do not rely on a raw prompt like:
- "What kind of company is TTWO?"

Use a hybrid design instead:

1. Deterministic feature extraction in code
- industry text
- margin shape
- revenue volatility
- capex intensity
- recurring revenue hints
- launch cadence
- franchise concentration
- filing/news keyword patterns

2. Rule-based scoring in code
- map features to candidate business-model labels

3. LLM as structured adjudicator
- consume features and candidate labels
- decide final primary/secondary business model
- explain why
- choose correct analytical framework

4. Framework registry in code
- each business model maps to a predefined analysis template

Example:
- `ip_driven`
  - emphasize release cadence, franchise durability, online tail, retention, monetization decay
- `saas`
  - emphasize recurring revenue, gross margin, retention, FCF conversion, operating leverage

## Prompt / Agent Design Principles

Do not use one monolithic prompt for everything.

Use chained, structured stages:

1. Fact extraction
- normalize raw payloads
- preserve provenance

2. Business model classification
- classify company type
- choose analytical framework

3. Interpretation
- turn metrics into economic meaning

4. Scenario generation
- build operational bull/base/bear trees

5. User adaptation
- tailor framing to long-term investor vs short-term trader

6. Final composition
- produce decisive investment memo

LLM role:
- reason over structured artifacts
- not act as the sole data layer, metric engine, and business model engine all at once

## Missing Data / Fallback Requirements

The system must never stop at "data missing".

Fallback rules:
- if financials fail:
  - use last cached statements
  - use SEC / filing fallback if available
  - use estimates if needed and label them clearly
- if ratios fail:
  - recompute locally from statements
- if event/news feed fails:
  - use cached event timeline
- if score endpoint fails:
  - compute locally from facts + interpretation + scenarios

Inference rules:
- infer from peers when appropriate
- use historical analogs when direct current metrics are unavailable
- distinguish observed vs inferred data in response

## TTWO / GTA VI-Specific Intelligence Requirements

System must be able to reason about:
- GTA V historical precedent
- GTA Online durability
- RDR2 as analog
- GTA VI launch impact
- launch spike vs long-tail monetization
- retention decay risk
- changing gamer behavior / attention fragmentation
- comparison to titles with weak post-launch online retention

The system should be able to answer:
- what happened after GTA V launch?
- how much of TTWO upside depended on GTA Online durability?
- could GTA VI repeat that outcome?
- what are the risks if GTA VI launches huge but engagement decays faster?

## Example Target Output Style

TTWO should not be summarized like a generic stock screener.

Instead, it should produce something closer to:
- TTWO is a non-linear IP asset, not a steady compounder
- GTA VI is the core catalyst, but the real question is long-tail monetization durability
- bull case depends on GTA VI becoming a persistent platform / online ecosystem
- bear case is that launch demand is enormous but retention and monetization decay faster than prior cycle
- conviction should separate franchise confidence from stock confidence

## Caching Strategy

Redis:
- quotes/news: short TTL
- normalized fact graph: medium TTL
- scenario outputs: medium TTL
- cached degraded-mode snapshots

Persistent store later:
- Postgres for normalized facts, event timelines, scenario artifacts, confidence history

Object storage later:
- raw provider payloads
- filings
- transcripts

Key principle:
- stale-but-usable with warning is better than empty or broken

## Guiding Principles

- Structure first, prompting second.
- Business-model-aware analysis is mandatory.
- Missing data should reduce confidence, not terminate reasoning.
- The system must distinguish facts, estimates, and inferences.
- The system should be useful even in degraded provider conditions.
- Output should be decision-oriented, not neutral summary text.
